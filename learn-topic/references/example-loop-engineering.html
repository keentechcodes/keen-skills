TL;DR — What it is

Core idea

An AI coding agent, stripped down, is "an LLM in a while loop with tools." A chatbot answers in one pass. An agent persists across many steps. Loop engineering is the discipline of designing the control system around that loop — the cadence, the tools, the verification, the memory, the stopping conditions, the guardrails — rather than hand-writing each prompt.

"You shouldn't be prompting coding agents anymore. You should be designing loops that prompt your agents."

— Peter Steinberger

"I don't prompt Claude anymore. I have loops running that prompt Claude and figure out what to do. My job is to write loops."

— Boris Cherny, Head of Claude Code, Anthropic

The shift — where the leverage moved

Layer 1
Prompt Engineering
shape the input to one model call
Layer 2
Context Engineering
shape what the model sees per turn
Layer 3
Harness Engineering
equip ONE agent run
Layer 4
Loop Engineering
orchestrate MANY runs over time
Harness
  • Scope: one agent run
  • Concerns: tools, context, permissions, rules
  • Lifetime: a session
  • Analogy: the sandbox
Loop
  • Scope: many runs over time
  • Concerns: scheduling, state, verification, handoff, escalation
  • Lifetime: days/weeks, survives restarts
  • Analogy: the factory floor

Each layer subsumes the previous. You still write prompts and curate context — but the leverage has moved one level up: from crafting individual prompts to designing the control system that schedules, verifies, and persists across many runs.

The loop cycle

Every loop, at every level, runs some variant of this cycle:

observe → decide → act → evaluate → update state → repeat

  P[Perceive

decide what to do]
  R --> A[Act
call a tool]
  A --> O[Observe
check the result]
  O -->|not done| P
  O -->|done| S([Stop
condition met])
1. Intent
define target outcome
2. Context
gather code, docs, errors
3. Action
edit files, run commands
4. Observation
capture results, diffs, logs
5. Adjustment
update plan, repeat
The single most-skipped stage is Observation/Verify. Skipping it collapses a loop back into a chain of one-shot calls.

The five primitives + memory

Cobus Greyling's reference framework. Every production loop is built from these six pieces.

1. Automations / Scheduling
The heartbeat — discovery + triage on a cadence
/loop (Grok), cron in Claude Code, GitHub Actions, /goal
2. Worktrees
Safe parallel execution — each agent gets its own working dir
isolation: "worktree" in Grok, --worktree, git worktrees
3. Skills
Persistent project knowledge that survives across runs
SKILL.md + scripts, CLAUDE.md, plugins
4. Plugins & Connectors (MCP)
Reach into real tools — tickets, PRs, DBs, dashboards
Model Context Protocol servers
5. Sub-agents
Maker/checker split — the writer never grades their own work
Explorer → Implementer → Verifier; separate model on /goal
+ Memory / State
The durable spine outside any conversation
STATE.md, LOOP-STATE.json, Linear board, GitHub Project, DB row
A minimal viable loop = Scheduling + one triage skill + a state file. Add the others only when the previous version has proven its value — and its failure modes.

The five design levers

The tunable knobs around the basic agent loop.

Lever What it controls Notes
ToolsWhat the agent can callScope tightly; least privilege
Stopping conditionsWhen the loop may quitSuccess, max-iterations, budget, no-progress detection
Context managementWhat survives across turnsCompaction, NOTES.md, sub-agent summaries (burn 30k, return 1k)
VerificationHow the loop proves it did the thingTests, linters, LLM-as-judge, JSON feature checklist with passes: false
GuardrailsWhat stops the loop doing harmSandboxes, scoped credentials, hard budgets (e.g. Willison's $5 Fly.io org)

A loop with no termination logic either declares victory early or never stops. A loop with no verification is a chain of calls pretending to be a loop.

The maker / checker split

Single most important pattern

The agent that wrote the code is a terrible judge of its own work.

flowchart LR
    Spec[Spec / Goal] --> Maker[Maker
implements]
    Maker --> Checker[Checker
independent model + rubric]
    Checker -->|pass| Done[Accept]
    Checker -->|fail + feedback| Maker
    Checker -->|stuck / ambiguous| Human[Human gate]
Common splits
  • Explorer → Implementer → Verifier
  • Implementer → Security reviewer
  • Implementer → Test writer + runner
Two verification models
  • Deterministic — test suites, linters, type checks. Binary pass/fail.
  • LLM-as-judge — second model scores output vs rubric. Use when "good" isn't binary.

/goal in Claude Code / Codex applies this to the stop condition itself: a fresh model decides whether the goal is met — not the one that did the work. This is the only reason you can walk away from an unattended loop with any confidence.

Steinberger's onboarding recipe

Start with dependency bumps, codemods, or flaky-test fixes. They are narrow in scope, have a binary pass/fail signal (CI green / diff clean), need no human judgement, and repeat on a schedule. Once one works reliably for a few weeks, the pattern transfers — but the trust is earned incrementally.

Concrete example — Daily Triage loop

The canonical first loop. Low cost, high learning.

sequenceDiagram
    participant Cron as ⏱ Cron (daily 9am)
    participant Triage as 👁 Triage Skill
    participant State as 📋 STATE.md
    participant Impl as ⚙️ Implementer
    participant Verify as ✓ Verifier
    participant PR as 🔗 GitHub PR
    participant Human as 🧑 You

    Cron->>Triage: fire
    Triage->>Triage: read yesterday's CI failures, open issues, recent commits
    Triage->>State: write findings
    alt nothing actionable
        Triage->>State: archive run, log "no-op"
    else actionable item
        State->>Impl: spawn in isolated worktree
        Impl->>Impl: draft fix
        Impl->>Verify: hand off diff
        Verify->>Verify: run tests + lint + spec check
        alt pass
            Verify->>PR: open PR
            PR->>Human: review request
        else fail
            Verify->>Impl: feedback, retry (max 3)
        end
    end
Week 1 command (Grok)
/loop 1d Run loop-triage. Update STATE.md. No auto-fix in week one.

That's it. Report-only. No auto-fix. Earn trust first.

Formal specification — LSS 1.0

The KanakMalpani/Loop-Engineering repo offers a formal spec format. Useful when loops graduate from "script in a TUI" to "production system with auditors."

A loop is formalized as L = (S, A, O, T, E, M, τ) — state, actions, observations, transitions, evaluators, memory, termination. 

loop_name: code-repair-loop
version: "1.0"
objective: "Fix failing tests with minimal diff"
workers:
  - role: implementer
    model: claude-sonnet
evaluators:
  - type: test_suite
feedback_channels:
  - id: test_failures_to_impl
    source: evaluators.test_suite
    destination: workers.implementer
termination_conditions:
  - type: all_tests_pass
  - type: max_iterations
    value: 10
cost_limits:
  max_tokens_per_run: 500000
safety_constraints:
  - no_force_push
  - no_destructive_migrations
Five conformance principles
  • Completeness
  • Boundedness (every loop must terminate)
  • Safety-first
  • Observability
  • Version discipline
D-D-M-I-S framework

The methodology that gives the spec teeth:

Design → Diagnose → Measure → Improve → Scale

Failure modes & anti-patterns

Loops amplify judgment — both good and bad
Infinite fix loops
Maker and checker ping-pong forever
Fix: cap iterations (3 attempts), then escalate to human
Verifier theater
Checker exists but never fails anything
Fix: independent model, real rubric, trap fixtures
Token burn
5-min loop runs full maker/checker every tick
Fix: triage cheap; spawn sub-agents only when state says actionable
Comprehension debt
Loop ships code faster than you read it
Fix: read what the loop made; the loop doesn't know what it doesn't know
Orphaned worktrees
Loop spawns, never cleans up
Fix: explicit cleanup step in every run
Auto-merge runaway
Allowlisted "safe" path turns out not to be
Fix: path denylist, auto-merge policy, MCP least privilege
Judgment atrophy
You stop having opinions; the loop runs you
Fix: design loops with judgment; using loops to avoid thinking is the accelerant

Two people can run the same loop and get opposite results. The loop doesn't know. You do.

Cost & safety defaults

Triage should be CHEAP — one model pass over signals, not maker+checker every tick.
Spawn sub-agents ONLY when state says actionable.
Cap iterations per item — 3 attempts, then escalate to human.
Give every loop a hard budget (tokens $$, run minutes).
Run report-only for one week before any auto-fix.

The loop-audit CLI scores your repo 0→100 on readiness: state file present, triage skill present, verifier skill, LOOP.md, safety docs, MCP configured, worktree isolation evidence, patterns/registry.yaml, loop-budget.md, loop-run-log.md, real run activity.

Checklist before shipping any loop

The honest caveats

  • Token costs explode with sub-agents and long-running loops. Budget before you schedule.
  • Verification is still on you. Unattended loops make unattended mistakes.
  • Comprehension debt grows faster than the codebase unless you read what the loop ships.
  • The loop doesn't know what it doesn't know. Two people run the same loop and get opposite results. You do.

Bottom line

Loop engineering is the runtime layer above prompt/context/harness engineering. Stop steering the agent turn-by-turn; design the system that schedules, assigns, verifies, persists, and escalates. Start with one cheap report-only loop on a task with a binary signal, run it a week, earn trust, then climb L1 → L2 → L3.

The loop is the lever; your judgment is the fulcrum.
